# OfflineTaxiRL

This project is an experiment that uses a taxi pickup-and-dropoff environment to explore offline reinforcement learning. The goal is to learn a policy from logged interaction data and compare offline RL against a behavior cloning baseline trained on the same state-action transitions.

The project uses multiple logged interaction datasets with different levels of data quality. This makes it possible to compare how behavior cloning and offline RL methods respond when the logs become less clean, less consistent, and more noisy.

The cleanest logs are generated by a hand-designed heuristic policy, which is described later in this README. Lower-quality logs are created by injecting random action noise into this heuristic and by mixing heuristic-generated episodes with random-policy episodes.

## How to Use

Run commands from the project root.

### 1. Generate logged interaction data

Generate whichever data quality levels you want to train on:

```bash
.venv/bin/python scripts/logs/generate_expert_logs.py
.venv/bin/python scripts/logs/generate_mild_noisy_logs.py
.venv/bin/python scripts/logs/generate_mixed_quality_logs.py
.venv/bin/python scripts/logs/generate_poor_noisy_logs.py
```

These commands create raw JSONL logs and transition pickle files in `data/`.

### 2. Train models

Use `--quality` to choose which logged dataset to train on. Valid quality values are:

```text
expert
mild
mixed
poor
```

Train behavior cloning:

```bash
.venv/bin/python scripts/train/train_bc.py --quality expert
.venv/bin/python scripts/train/train_bc.py --quality mild
.venv/bin/python scripts/train/train_bc.py --quality mixed
.venv/bin/python scripts/train/train_bc.py --quality poor
```

Train CQL:

```bash
.venv/bin/python scripts/train/train_cql.py --quality expert
.venv/bin/python scripts/train/train_cql.py --quality mild
.venv/bin/python scripts/train/train_cql.py --quality mixed
.venv/bin/python scripts/train/train_cql.py --quality poor
```

Train BCQ:

```bash
.venv/bin/python scripts/train/train_bcq.py --quality expert
.venv/bin/python scripts/train/train_bcq.py --quality mild
.venv/bin/python scripts/train/train_bcq.py --quality mixed
.venv/bin/python scripts/train/train_bcq.py --quality poor
```

Trained models are saved with quality-specific names, for example:

```text
data/bc_model_mixed.pt
data/offline_rl_cql_mixed.d3
data/offline_rl_bcq_mixed.d3
```

### 3. Evaluate models

Evaluate a behavior cloning model:

```bash
.venv/bin/python scripts/evaluate_bc.py --quality mixed
```

If `--quality` is omitted, the evaluator uses the first `data/bc_model_*.pt` model it can find.

Evaluate an offline RL model:

```bash
.venv/bin/python scripts/evaluate_offline_rl.py --algorithm cql --quality mixed
.venv/bin/python scripts/evaluate_offline_rl.py --algorithm bcq --quality mixed
```

If `--algorithm` is omitted, the evaluator looks for a CQL model first and then a BCQ model. If `--quality` is omitted, it uses the first matching model file it can find.

## Taxi Environment

The environment is a grid world where a taxi moves one square at a time using four discrete actions: up, left, down, and right.

Each episode includes passengers with:

- a pickup location
- a dropoff location
- a payout for successful delivery

The taxi must:

- navigate to a passenger
- pick them up
- carry them to the correct dropoff point

The environment is fully observable, deterministic, and sequential, which makes it a useful small-scale testbed for comparing imitation learning and offline RL methods.

## Heuristic Policy Scoring

The log-generation policy chooses which passenger to target with this score:

`score = payout - ALPHA_PICKUP * pickup_distance - BETA_TRIP * trip_distance`

### What each term means

- `payout`: the reward for successfully delivering that passenger.
- `pickup_distance`: the Manhattan distance from the taxi's current position to the passenger's pickup location.
- `trip_distance`: the Manhattan distance from that passenger's pickup location to their dropoff location.

### Why `ALPHA_PICKUP` and `BETA_TRIP` exist

- `ALPHA_PICKUP` controls how much the policy penalizes going far just to reach a passenger.
- `BETA_TRIP` controls how much the policy penalizes choosing a passenger whose ride will be long after pickup.

These are hand-tuned weights. They let the heuristic balance three competing goals:

- prefer passengers with higher payout
- prefer passengers that are closer to reach
- prefer passengers whose full trip is not too expensive in steps

In this project:

- `ALPHA_PICKUP = 0.8`
- `BETA_TRIP = 0.35`

That means pickup distance is penalized more heavily than trip distance. The heuristic is therefore more sensitive to how far the taxi must travel before service can even begin.

### How the score is used

The score is recomputed at every step while the taxi is not carrying a passenger.

- The heuristic evaluates every waiting passenger.
- It picks the passenger with the highest score.
- It moves one step toward that passenger's pickup point.

Once a passenger has been picked up, the policy stops rescoring candidates and commits to that passenger's dropoff location until the ride is completed.

## Logged Data Quality Levels

The experiments use four levels of logged interaction data. All levels use the same taxi environment, state representation, action space, and reward function. They differ only in how the action is selected during data generation.

### Expert logs

Expert logs are generated entirely by the best-value heuristic policy described above.

- policy: `100%` best-value heuristic
- random action noise: `0%`
- transition file: `data/transitions_best_value.pkl`
- raw log file: `data/raw_taxi_logs_best_value.jsonl`

These logs represent the cleanest state-action mapping. In these data, similar states should usually produce similar actions.

### Mild noisy logs

Mild noisy logs mostly follow the same best-value heuristic, but a random valid movement action is selected on some steps.

- policy: best-value heuristic with action noise
- random action noise: `10%`
- transition file: `data/transitions_mild_noisy.pkl`
- raw log file: `data/raw_taxi_logs_mild_noisy.jsonl`

These logs represent a mostly competent policy with occasional mistakes. They are intended to test whether a model can still learn when the demonstrated behavior is slightly inconsistent.

### Mixed-quality logs

Mixed-quality logs combine episodes from several behavior sources.

- `50%` best-value heuristic episodes
- `30%` noisy best-value heuristic episodes with `30%` random action noise
- `20%` random-policy episodes
- transition file: `data/transitions_mixed_quality.pkl`
- raw log file: `data/raw_taxi_logs_mixed_quality.jsonl`

These logs represent heterogeneous logged behavior. The dataset still contains many good trajectories, but it also contains inefficient and inconsistent actions.

### Poor noisy logs

Poor noisy logs contain a smaller amount of clean heuristic behavior and a larger amount of noisy or random behavior.

- `20%` best-value heuristic episodes
- `30%` noisy best-value heuristic episodes with `50%` random action noise
- `50%` random-policy episodes
- transition file: `data/transitions_poor_noisy.pkl`
- raw log file: `data/raw_taxi_logs_poor_noisy.jsonl`

These logs represent the lowest-quality dataset. They are useful for testing whether offline RL can still use reward information to recover useful behavior when behavior cloning is more likely to imitate bad actions directly.

## Reward Design

The project uses a simple step-based reward design:

- valid move: `-0.1`
- invalid move: `-1.0`
- successful dropoff: add the passenger payout

### Why this reward exists

- The small negative reward for a valid move encourages shorter, more efficient routes.
- The larger negative reward for an invalid move discourages wasteful actions such as pushing into grid boundaries.
- The positive dropoff reward makes successful task completion the main objective.

### Reward formula per step

For a step with no successful dropoff:

- valid move -> `-0.1`
- invalid move -> `-1.0`

For a step with a successful dropoff:

- reward = move penalty + passenger payout

Example:

- if the taxi makes a valid move and drops off a passenger with payout `12`
- step reward = `-0.1 + 12 = 11.9`
